# 🎰 로또 시스템 (순수 자바 TDD)

## 📌 개요
비즈니스 룰을 객체 내부에 안전하게 캡슐화하고, 테스트하기 어려운 로직(Random)을 분리하는 역량을 평가합니다.

## 🎯 요구사항
1. **로또 구매**
   - 구입 금액을 입력하면 1,000원 단위로 로또를 자동 발급합니다.
   - 각 로또는 1~45 사이의 중복되지 않는 6개의 숫자를 가집니다.
2. **당첨 확인 및 통계**
   - 지난 주 당첨 번호(6개)와 보너스 번호(1개)를 입력받습니다.
   - 발행된 로또들과 비교하여 일치하는 번호 개수(3개~6개)를 구합니다.
   - 총 당첨 금액을 합산하고, 구매 금액 대비 **총 수익률**을 계산하여 반환합니다.

## 💡 평가 포인트 (리뷰 기준)
- 로또 번호(`LottoNumber`), 로또 1장(`Lotto`)을 **불변 객체**이자 **일급 컬렉션**으로 포장했는가?
- 무작위 번호를 추출하는 `Random` 로직을 인터페이스로 추상화하여, 단위 테스트 시 원하는 번호를 고정으로 뽑을 수 있도록(Mock/Fake) 제어했는가?

## 배운점
- 1. 파라미터가 3~4개씩 계속 뭉쳐 다니네? 변수들 끼리 뭔가 관계식이 있는데 -> 응집의 원리 `WinningLotto`
- 2. 생성자 안이 너무 뚱뚱한데? 자기가 태어나는 방법이 너무 복잡하다 테스트 하기 빡세다 -> 생성책임의 분리 `LottoMachine`
- 3. 도메인 안에서 `Random` 직접 돌리니까 테스트 불편하네? -> 의존성 역전 원리
- 4. `subList()`로 자른 결과를 바로 반환해도 되나? -> 메모리 누수 방지 및 원본 변형 차단 방어적 복사
- 5. 여러 장의 로또(`List<Lotto>`)를 까보면서 당첨결과 집계하는 책임은 누가 가지지? -> 일급 컬레션 `LottoTicket`, `LottoResult` 

#### 💡 로또 도메인 구현 핵심 포인트 정리
*   **일급 컬렉션 활용 (`Lotto`)**: `List<Integer>` 원시 컬렉션을 포장하여, 로또 번호의 6개 여부, 중복 여부, 범위 검증 책임을 도메인 내부로 캡슐화했습니다.
*   **비즈니스 로직의 Enum 분리 (`Rank`)**: 일치하는 개수와 보너스 번호 여부를 받아 당첨 등급(상금)을 결정하는 복잡한 조건문(if-else)을 Enum 내부로 위임하여 객체지향적으로 풀어냈습니다.
*   **불변성 보장**: `getNumbers()` 반환 시 `Collections.unmodifiableList()`를 사용하여 외부에서 객체의 상태를 임의로 조작할 수 없도록 방어했습니다.
