# 🚗 주차장 시스템 (순수 자바 OOP)

## 📌 개요
프레임워크(Spring) 없이 순수 자바의 객체지향적 설계와 자료구조 활용 능력을 평가합니다.

## 🎯 요구사항
1. **차량 종류 및 전용 주차 구역**
   - 차량은 종류(예: `SMALL`(소형), `MEDIUM`(중형), `LARGE`(대형) 등)를 가집니다.
   - 주차장 역시 각 차량 종류에 맞는 전용 주차 칸(Spot)들을 배정받아 생성됩니다. (예: 소형 전용 10칸, 대형 전용 5칸 등)
2. **차량 입차 (Park)**
   - 차량이 들어오면 해당 차량의 종류가 주차할 수 있는 전용 '빈자리'를 확인합니다.
   - 빈자리가 있으면 주차권(`Ticket`)을 발급합니다. 주차권에는 '차량 번호', '입차 시간', '배정된 주차 칸 정보'가 기록됩니다.
   - 해당 차량 종류의 빈자리가 없다면 `IllegalArgumentException` 등 예외를 발생시킵니다.
3. **차량 출차 (Leave)**
   - 주차권과 출차 시간을 입력받아 주차장에서 해당 차량을 빼내고 칸을 다시 빈자리로 만듭니다.
   - 주차 시간(출차 시간 - 입차 시간)을 계산하여 요금을 부과합니다. (기본: 시간당 1,000원, 올림 처리. 단, 차량 종류별로 다른 요금 정책을 적용할 수 있다면 객체지향적 설계 가산점)
4. **주차장 현황 조회 (Status)**
   - **주차 가능한 칸 수**: 전체 혹은 차량 종류별로 현재 주차 가능한 빈자리 개수를 조회할 수 있어야 합니다.

## 💡 평가 포인트 (리뷰 기준)
- 다형성(Polymorphism)이나 Enum을 활용하여 차량 종류별 검증 및 주차 로직을 깔끔하게 분리했는가? (If-else 분기 최소화)
- `Map`, `Set`, `List` 등 적절한 컬렉션을 사용하여 빠르고 효율적으로 빈자리를 찾고 차량을 관리할 수 있는가?
- 주차 칸(`ParkingSpot`)이라는 객체를 새로 도출하여 상태(점유 여부, 차량 종류, 번호)를 스스로 관리하도록 캡슐화했는가?
- 테스트 코드 작성을 고려하여 `LocalDateTime` 같은 시간 객체를 외부에서 주입받도록 설계했는가?


## 풀면서 생각한점
- 1. `ParkingSpot`도 `Car`를 가지고 `ParkingLot`도 주차정보 가진다.
  - 하는게 맞다. 차량이 들어오고 나갈때 마다 변화하도록 하자. 인덱스와 객체지향의 개념, 동기화 주의할것
- 2. 컬렉션 자료구조를 선택할땐 복잡도를 반드시 유의할것
  - `List`는 O(N)의 시간복잡도 나올 수 있다. 요소 커질수록 비효율
  - `Map`은 O(1)의 시간복잡도 가진다. 대신 key를 무엇으로 할지 고민이 필요함 (무엇을 찾기 위해 쓰는가?)
    - 투 트랙 관리전략 (빈자리 그룹, 이용자 그룹) - 주차되지 않은 칸도 관리가 필요함
      - `Map<String, ParkingSpot>` 이용중인 자리이다. 번호판으로 관리
      - `Map<Type, Queue<ParkingSpot>>` 빈자리 찾아줄땐, 맨앞에서 빠르게 차 빠질때도 뒤에 빠르게 추가. 타입으로 관리 
- 3. 주차장의 초기 제한 대수를 어떻게 주입 받을 것인가? (OCP 원칙 적용)
  - 새로운 타입이 추가될때 마다 생성자를 수정하는 상황을 막고싶다. `Map<Type, Integer>`한번에 주입받자
  - 스스로 입력받은 `Map`기반으로 내부 큐를 채우도록 **객체 생성의 책임을 주차장 내부로 숨긴다.**
  - 생성자는 외부에서 필요한 값을 줄뿐, 어떻게 저장할지 가공은 객체 내부의 책임이다.
- 4. `Ticket` 사건의 발생 이벤트를 객체로 묶을것인가? 
  - 주차 공간(`ParkingSpot`)의 책임은 물리적 공간 관리. 요금 정산이나 시간 기록까지 하면 객체가 너무 무거워짐.
  - 이력(History) 및 정산 관리를 위해 거래/이벤트(Transaction)를 나타내는 `Ticket` 객체로 책임을 분리 (SRP 원칙).

## 🚀 Day 1 핵심 회고 (객체지향 5대 무기)
1. **O(1) 자원 풀(Pool) 아키텍처**: 투 트랙 전략(이용자 Map, 빈자리 Queue)으로 대규모 주차장에서도 탐색 속도를 `O(1)`로 최적화.
2. **유연한 생성자 (OCP 원칙)**: `Map<Type, Integer>` 주입으로, 미래에 새로운 차종이 추가되더라도 기존 코드를 수정할 필요가 없음.
3. **능동적인 객체 (캡슐화)**: 단순 Setter를 지양하고, `ParkingSpot` 스스로 `enterCar()`, `leaveCar()`를 통해 상태를 방어하고 예외를 던지도록 설계.
4. **역할과 책임 분리 (SRP)**: 공간의 상태 정보(`ParkingSpot`)와 거래/매출 정보(`Ticket`)를 분리하여 시스템의 확장성을 확보.
5. **메시지 위임 (다형성)**: 주차장이 직접 요금을 계산(`if-else`)하지 않고, 요금 정보를 가진 `Type` Enum에게 계산 책임을 위임.
