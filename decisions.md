# [VMS 과제] S2. 선원 익명 커뮤니티 게시판 - 설계 의사결정 내역 (decisions.md)

## 1. 익명 닉네임 부여 방식 및 충돌 방지 (동시성 제어)
* **요구사항:** 사용자는 게시글 단위로 고유한 익명 닉네임을 부여받아야 하며, 도메인 특성(해양산업)을 반영해야 함.
* **대안 A (선택):** '형용사 + 해양생물 명사'의 무작위 조합 (예: 푸른 + 돌고래).
* **대안 B:** 순차적 식별자 발급 (예: 익명1, 익명2...).
* **선택 이유:** 대안 B가 동시성 제어에 유리하나, 커뮤니티의 해양 도메인 컨셉을 살리고 사용자 경험(UX)을 향상시키기 위해 대안 A를 선택함.
* **트레이드오프 및 한계 극복:**
  * 다수 사용자가 동시에 첫 댓글을 작성할 때 닉네임 중복(Check-Then-Act 동시성 문제) 및 해시 충돌(Birthday Paradox)이 발생할 수 있음.
  * 이를 해결하기 위해 애플리케이션 락이 아닌 데이터베이스 복합 유니크 제약조건(`UNIQUE(post_id, nickname)`)을 활용함. `DataIntegrityViolationException` 발생 시 최대 3회 재시도(Retry)하며, 4회째부터는 닉네임 뒤에 3자리 난수를 붙여 강제로 유니크를 보장하고 무한 루프를 방지함.

## 2. 익명 닉네임 관리 테이블 설계 및 무결성 보장
* **대안 A:** `comments` 테이블에 `nickname` 컬럼을 직접 추가.
* **대안 B (선택):** `post_nickname_mappings` 테이블을 별도로 분리하여 관리.
* **선택 이유:**
  * 대안 A는 동시성 제어가 어렵고, 한 사용자가 여러 댓글을 달 경우 데이터 중복이 발생함.
  * 별도 매핑 테이블을 구성하고 `UNIQUE(post_id, user_id)` 제약조건을 걸어, DB 레벨에서 완벽하게 '1게시글 1인 1닉네임'을 보장함. 작성자 역시 본 매핑 테이블에 기록하여 일관된 닉네임 조회 아키텍처를 확보함.

## 3. 물리적 외래키(FK) 제약 조건 제거 (DB 락 최적화 및 Soft Delete 고려)
* **고민:** 대용량 쓰기 트래픽 환경 및 댓글의 Soft Delete 요구사항에 대응하기 위한 스키마 설계.
* **대안 A:** 데이터베이스 레벨의 물리적 외래키(`CONSTRAINT FOREIGN KEY`) 제약 유지.
* **대안 B (선택):** 물리적 외래키 제약을 제거하고, 논리적 외래키(조회용 인덱스 추가)와 애플리케이션 레벨의 무결성 검증으로 대체.
* **선택 이유 및 트레이드오프:**
  * 대안 A는 완벽한 참조 무결성을 보장하지만, PostgreSQL의 `FOR KEY SHARE` 락 특성상 초고트래픽 쓰기 상황에서 부모 레코드(게시글)에 대한 Lock 큐잉 현상을 유발해 DB 병목이 됨.
  * 대안 B를 선택하여 쓰기 성능을 확보함. 단, 참조 무결성 훼손을 방어하기 위해 **모든 Write(댓글/반응) 요청 시 서비스 레이어에서 타겟 레코드의 존재 여부 및 `is_deleted = false` 상태를 반드시 검증**하도록 강제함.

## 4. 게시글 목록 카운트(조회수/좋아요/댓글) 쿼리 최적화
* **고민:** 목록 조회 시 통계 데이터를 어떻게 빠르고 정확하게 가져올 것인가?
* **대안 A:** 조회 시점마다 `COUNT()` 실시간 수행.
* **대안 B (선택):** `posts` 테이블에 카운트 컬럼을 역정규화(Denormalization)하고 원자적 쿼리로 업데이트.
* **선택 이유 및 트레이드오프:**
  * 데이터가 늘어날수록 `COUNT()`는 심각한 조회 지연(O(N))을 유발함. Read 비율이 높은 커뮤니티 특성상 대안 B를 선택해 읽기 성능을 O(1)로 극대화함.
  * 쓰기 시점의 갱신 손실(Lost Update) 방지를 위해 트랜잭션 내에서 원자적 업데이트(`UPDATE posts SET view_count = view_count + 1 WHERE id = ?`)를 수행하여 정합성을 보장함.

## 5. 반응(좋아요/싫어요) 토글 로직과 상태 전이(State Transition)
* **고민:** 반응 취소/변경 시 발생하는 동시성 문제와 카운트 정합성 유지.
* **대안 A:** 레코드 물리적 삭제(`DELETE`) 활용.
* **대안 B (선택):** UPSERT 구문을 활용하되, 반응 취소 시 레코드를 삭제하지 않고 `NULL`로 상태만 변경.
* **선택 이유 및 트레이드오프:**
  * 반응이 자주 변경되는 환경에서 잦은 `DELETE/INSERT`는 DB 인덱스 단편화(Fragmentation)를 유발함. 취소 시 `NULL`로 업데이트하여 유니크 키를 유지하고 I/O 비용을 절감함.
  * **카운트 정합성:** UPSERT 실행 시 이전 상태를 기준으로 게시글의 `like_count`, `dislike_count`를 증감시키는 '상태 전이 규칙'을 트랜잭션 내에 엄격하게 정의하여 캐시 불일치를 원천 차단함.

## 6. 커서 기반 페이지네이션(Cursor-based Pagination) 도입
* **고민:** 대용량 게시판에서 스크롤/페이징을 처리하는 최적의 방법.
* **대안 A:** `offset` 기반 방식.
* **대안 B (선택):** 마지막 조회 식별자(`last_id`) 기반 커서 방식.
* **선택 이유 및 트레이드오프:**
  * 대안 A는 데이터가 커질수록 검색 비용이 선형 증가하며, 조회 중 새 글이 등록되면 중복 노출 문제가 발생함.
  * 대안 B는 인덱스(`idx_channel_id`)를 통한 Range Scan으로 O(1) 성능을 유지함. 임의 페이지 Jump는 불가하지만 무한 스크롤 환경에 적합함.
