# [VMS 과제] S2. 선원 익명 커뮤니티 게시판 - 설계 의사결정 내역 (decisions.md)

## 1. 익명 닉네임 부여 방식 및 충돌 방지 (동시성 제어)
* **요구사항:** 사용자는 게시글 단위로 고유한 익명 닉네임을 부여받아야 하며, 도메인 특성(해양산업)을 반영해야 함.
* **대안 A (선택):** '형용사 + 해양생물 명사'의 무작위 조합 (예: 푸른 + 돌고래).
* **대안 B:** 순차적 식별자 발급 (예: 익명1, 익명2...).
* **선택 이유:** 대안 B가 동시성 제어에 유리하나, 커뮤니티의 해양 도메인 컨셉을 살리고 사용자 경험(UX)을 향상시키기 위해 대안 A를 선택함.
* **트레이드오프 및 한계 극복:**
    * 다수 사용자가 동시에 첫 댓글을 작성할 때 닉네임 중복(Check-Then-Act 동시성 문제) 및 해시 충돌(Birthday Paradox)이 발생할 수 있음.
    * 이를 해결하기 위해 애플리케이션 락이 아닌 데이터베이스 복합 유니크 제약조건(`UNIQUE(post_id, nickname)`)을 활용함. `DataIntegrityViolationException` 발생 시 최대 3회 재시도(Retry)하며, 4회째부터는 닉네임 뒤에 3자리 난수를 붙여 강제로 유니크를 보장하고 무한 루프를 방지함.

## 2. 익명 닉네임 관리 테이블 설계 및 무결성 보장
* **대안 A:** `comments` 테이블에 `nickname` 컬럼을 직접 추가.
* **대안 B (선택):** `post_nickname_mappings` 테이블을 별도로 분리하여 관리.
* **선택 이유:**
    * 대안 A는 동시성 제어가 어렵고, 한 사용자가 여러 댓글을 달 경우 데이터 중복이 발생함.
    * 별도 매핑 테이블을 구성하고 `UNIQUE(post_id, user_id)` 제약조건을 걸어, DB 레벨에서 완벽하게 '1게시글 1인 1닉네임'을 보장함. 작성자 역시 본 매핑 테이블에 기록하여 일관된 닉네임 조회 아키텍처를 확보함.

## 3. 물리적 외래키(FK) 제약 조건 제거 (DB 락 최적화 및 Soft Delete 고려)
* **고민:** 대용량 쓰기 트래픽 환경 및 댓글의 Soft Delete("삭제된 댓글입니다" 처리) 요구사항.
* **설계 결정:** 물리적 외래키(Constraint Foreign Key) 제약을 제거하고, 논리적 외래키(인덱스)와 애플리케이션 레벨의 무결성 검증으로 대체.
* **선택 이유:**
    * **PostgreSQL의 락 메커니즘 한계 고려:** PostgreSQL은 FK 검증 시 `FOR KEY SHARE` 락을 사용함. 이는 동시성이 뛰어나지만, 초고트래픽으로 인해 다수의 트랜잭션이 동시에 동일 게시글에 댓글을 `INSERT` 할 경우 부모 레코드(post)에 대한 Lock 큐잉 현상이 발생하여 DB I/O 병목이 됨.
    * **Soft Delete(논리적 삭제) 관리:** 자식(답글)이 있는 부모 댓글의 상태 변경 시 물리적 참조 무결성 규칙과 충돌하는 것을 방지함. 참조 무결성 훼손 위험은 Spring Boot 애플리케이션 레이어의 비즈니스 로직으로 방어함.

## 4. 게시글 목록 카운트(조회수/좋아요/댓글) 쿼리 최적화
* **고민:** 목록 조회 시 매번 `COUNT()` 쿼리를 수행할 것인가, 역정규화(Denormalization)할 것인가?
* **설계 결정:** `posts` 테이블에 `view_count`, `like_count`, `comment_count` 컬럼을 역정규화하여 관리.
* **선택 이유:**
    * 많은 데이터 환경에서 다중 테이블 JOIN 및 `COUNT()` 연산은 조회 지연을 유발함. Read 비율이 압도적으로 높은 커뮤니티 특성상 쓰기 시점의 비용을 감수하고 읽기 성능을 극대화(O(1))하는 것이 적합함.
    * 비동기 이벤트 버스 사용이 제한된 본 과제 환경에 맞춰, 쓰기 트랜잭션 발생 시 DB의 원자적 업데이트 쿼리(`UPDATE posts SET comment_count = comment_count + 1 WHERE id = ?`)를 사용하여 락 경합을 최소화하며 정합성을 맞춤.

## 5. 반응(좋아요/싫어요) 토글 로직 설계
* **고민:** "좋아요 → 다시 좋아요 = 취소", "좋아요 → 싫어요 = 변경" 로직 처리.
* **설계 결정:** UPSERT (PostgreSQL의 `ON CONFLICT DO UPDATE`) 구문 활용.
* **선택 이유:** 조회 후 `INSERT/UPDATE/DELETE` 분기를 태우는 Check-Then-Act 로직은 동시성 이슈를 유발함. DB의 `UNIQUE(target_type, target_id, user_id)` 제약을 기반으로 UPSERT를 사용하여 한 번의 쿼리로 원자적 상태 변경을 수행함.

## 6. 커서 기반 페이지네이션(Cursor-based Pagination) 도입
* **고민:** `page`와 `offset`을 사용하는 전통적인 방식과 `cursor` 기반 방식 중 어떤 것을 선택할 것인가?
* **설계 결정:** 성능 최적화를 위해 오프셋 방식 대신 커서 기반 페이지네이션을 채택함.
* **선택 이유:**
  * **조회 성능 최적화 (O(N) -> O(1)):** 오프셋 방식(`OFFSET N LIMIT M`)은 N이 커질수록 DB가 앞선 N개의 레코드를 스캔해야 하므로 데이터 양이 많아질수록 응답 속도가 기하급수적으로 느려짐. 반면, 마지막 조회 ID를 기준으로 조회하는 커서 방식은 인덱스를 타고 즉시 타겟 지점으로 이동하므로 대용량 데이터 환경에서도 일정한 성능을 유지함.
  * **데이터 정합성 유지:** 사용자가 목록을 보는 사이에 새로운 게시글이 등록되어도 데이터가 밀려 중복 노출되거나 누락되는 현상을 원천 차단함.
* **트레이드오프:** 상대적으로 구현이 복잡하지만, 무한 스크롤이나 '더보기' 형태의 커뮤니티 UI 특성상 성능 이득이 훨씬 크다고 판단함.
